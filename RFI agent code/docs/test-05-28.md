## Bugs Found & Fixed

### Bug 1: `category_hint` field never read (CRITICAL)
- **Root Cause**: `match_and_fill()` used `q.get("category", "Uncategorized")` but the parser stores the value as `category_hint`. Every question resolved to "Uncategorized".
- **Impact**: Zero base_info was ever passed to the LLM → all answers came back empty/0% confidence.
- **Fix**: Changed to `q.get("category_hint") or q.get("category", "Uncategorized")` in both sync and async paths.
- **Tests**: #1, #2, #13

### Bug 2: No fallback when category doesn't match `category_to_base` map (CRITICAL)
- **Root Cause**: When category was "Uncategorized" or any unmapped value, the code iterated over an empty list → `base_info_text` stayed empty → LLM prompt said "No base info available".
- **Impact**: The LLM was told not to hallucinate, received zero context, and correctly returned nothing. User saw all-blank answers.
- **Fix**: When `category_to_base.get(category)` returns empty, include ALL base_info files (~54KB, fits easily in context).
- **Tests**: #3, #4, #5

### Bug 3: Whitespace-only `question_text` leaks to LLM (LOW)
- **Root Cause**: `if not question_text:` is `False` for `"   "` (whitespace is truthy in Python).
- **Impact**: Wasted LLM call, potentially nonsensical answer.
- **Fix**: Added `.strip()` before the check.
- **Tests**: #11

## Glaring Issues Identified (High-Level Review)

1. **Parser column detection wrong for Pfizer file** — The Pfizer `.xlsm` has an explicit "CATEGORY" column (col A) and "QUESTION" column (col C), but the parser sometimes reads the RESPONSE column (col D) as questions. The heuristic detection works for most files but fails on this layout.

2. **No semantic search in local fallback** — `_find_similar_qas_json()` does exact category match only, no fuzzy/keyword matching. A question about "employees" in an unrecognized category won't find the "People Information" entries in the KB.

3. **54KB context per question** — Including ALL base_info for every unmatched question works but burns tokens. When Azure Search is enabled, this won't be an issue (semantic chunking returns only relevant sections). For local dev, it's fine.

4. **`review_answers` groups by exact category string** — If two questions have slightly different category hints (e.g. "Compliance" vs "compliance"), they won't be reviewed together.
